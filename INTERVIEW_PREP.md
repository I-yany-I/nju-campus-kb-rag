# 南京大学校园办事指南 RAG 面试准备

这份材料用于面试前快速理解和复述本项目。建议先掌握“项目要解决什么问题、为什么这样设计、每个模块做什么、指标怎么解释、有什么不足和改进方向”，再背后面的问答。

## 1. 项目一句话介绍

南京大学校园办事指南 RAG 问答系统，是一个面向南大学生常见 IT、教务、学生服务、财务、出国境办事问题的文本知识库问答项目。系统把校园办事材料整理成 JSONL 知识库，通过 BM25 关键词召回、Sentence-Transformer 向量召回、RRF 融合、BERT Cross-Encoder 重排找到证据片段，再基于证据生成带引用的中文回答；当证据不足时主动拒答，避免大模型凭记忆编造学校政策。

面试时可以这样说：

> 这个项目不是单纯调用大模型问答，而是做了一个可引用、可拒答、可评估的校园办事 RAG 系统。核心目标是把学校办事指南变成可检索的知识库，并让回答始终落在可追溯证据上。

## 2. 项目背景与痛点

学生办理校园事务时常见问题包括 VPN 怎么用、统一身份认证密码怎么找回、成绩单/在读证明在哪里办、选课退课规则是什么、医保和就业手续怎么处理等。这些信息分散在信息化中心、本科生院、图书馆、校医院、就业中心等多个站点中，用户很难快速定位。

如果直接让通用大模型回答，会有两个问题：

- 学校政策具有时效性，模型训练语料可能过期。
- 大模型可能把常识或其他学校规则当作南京大学规则，产生幻觉。

所以本项目采用 RAG：先检索项目内知识库的可信片段，再基于片段回答，并把来源一起返回。

## 3. 数据与知识库设计

知识库位于 `data/campus_kb/nju_official_kb.jsonl`，共 75 条文档。每条记录是一行 JSON，主要字段包括：

- `id`：文档唯一编号，如 `nju-it-vpn`。
- `title`：标题，如“校园 VPN 与校外访问说明”。
- `department`：主管部门。
- `source`：官方入口或栏目页链接。
- `updated_at` / `collected_at`：材料更新时间和收集时间。
- `tags`：便于检索的关键词。
- `text`：办事说明正文。

这些条目覆盖六类场景：`it`、`academic`、`student`、`finance`、`international`、`refusal`。其中 `refusal` 类用于帮助系统识别知识库边界，例如实时信息、个人账号查询、隐私信息等不应回答的问题。

为什么使用 JSONL？

- 方便增量追加，一条文档一行。
- 易于脚本读取和版本控制。
- 每条文档可以携带完整元数据，便于引用和评估。

## 4. 系统整体流程

完整链路如下：

```text
用户问题
  -> 问题标准化
  -> BM25 关键词召回
  -> Sentence-Transformer 向量召回
  -> RRF 融合候选
  -> BERT Cross-Encoder 重排
  -> 低置信度 / 拒答文档过滤
  -> 抽取式回答或 LLM 生成
  -> 返回答案、引用来源、检索片段
```

对应核心文件：

- `config/campus_kb.yaml`：集中配置知识库、索引、检索、生成和拒答阈值。
- `src/campus_kb_rag/documents.py`：加载 JSONL 并做滑动窗口分块。
- `src/campus_kb_rag/retriever.py`：构建 FAISS 索引、BM25 索引、混合检索和 Cross-Encoder 重排。
- `src/campus_kb_rag/generator.py`：抽取式回答、Prompt 构造、LLM/LoRA 可选生成。
- `src/campus_kb_rag/pipeline.py`：端到端封装 `ask()`。
- `build_campus_kb_index.py`：构建索引。
- `evaluate_campus_kb.py`：离线评估。
- `app.py`：Gradio 演示界面。

## 5. 模块细节

### 5.1 文档加载与分块

`documents.py` 先读取 JSONL，将每条记录转成 `KBDocument`，再用滑动窗口切成 `KBChunk`。当前配置是：

- `chunk_size: 420`
- `chunk_overlap: 80`

分块原因：

- 办事说明可能较长，直接整篇检索会稀释局部相关性。
- 分块后可以让向量表示更聚焦。
- overlap 可以减少答案信息被切断的问题。

面试表达：

> 我没有直接把整篇文档喂给检索，而是用固定长度加重叠窗口切块。这样能提高局部匹配效果，也能保证引用片段足够短，方便展示给用户。

### 5.2 稠密向量召回

稠密召回使用 `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`。构建索引时会把每个 chunk 的标题、部门、标签和正文拼接后编码，再用 FAISS `IndexFlatIP` 建索引。

代码中会对 embedding 做 L2 normalize，因此内积近似等价于余弦相似度。

优点：

- 能处理同义表达，例如“校外访问数据库”和“VPN 怎么用”。
- 对中文自然语言问题更鲁棒。

不足：

- 可能漏掉强关键词，如具体系统名、证件名。
- 依赖模型缓存和 Hugging Face 下载环境。

### 5.3 BM25 关键词召回

BM25 使用 `rank_bm25`。中文没有复杂分词器，项目中采用了简单 tokenizer：

- 英文、数字、下划线按正则抽取。
- 中文按单字和 bigram 提取。

为什么需要 BM25？

- 对“VPN”“GPA”“CET”“Office”“MATLAB”等关键词很敏感。
- 可以弥补向量模型对短词、专有名词不稳定的问题。

### 5.4 RRF 融合

项目用 RRF（Reciprocal Rank Fusion）融合稠密召回和 BM25 召回。公式可以理解为：

```text
score(doc) += 1 / (rrf_k + rank + 1)
```

当前 `rrf_k` 是 60。RRF 不直接比较 BM25 分数和向量相似度，而是比较排序位置，因此适合融合不同检索器。

面试表达：

> BM25 和向量召回的分数尺度不一样，直接相加不合理。我用 RRF 按排名融合，避免手动调两个检索器的分数权重。

### 5.5 BERT Cross-Encoder 重排

初步召回后，项目用 `cross-encoder/mmarco-mMiniLMv2-L12-H384-v1` 对 `(问题, 片段)` 做精排。

Bi-Encoder 和 Cross-Encoder 的区别：

- Bi-Encoder：问题和文档分别编码，适合大规模召回，速度快。
- Cross-Encoder：问题和文档一起输入模型，能建模细粒度交互，排序更准，但速度慢。

本项目先召回，再重排，只对前 10 个候选做 Cross-Encoder，因此能兼顾速度和精度。

### 5.6 低置信度拒答

`pipeline.py` 中有 `_filter_low_confidence()`：

- 如果没有检索结果，拒答。
- 如果第一条命中的是拒答边界文档 `nju-support-unknown`，拒答。
- 如果 top score 低于 `prompt.refusal_threshold`（当前 0.18），拒答。

拒答的价值：

- 避免回答知识库没有覆盖的问题。
- 避免把实时信息、个人隐私、账号查询类问题包装成确定答案。
- 面向校园政策类场景更安全。

### 5.7 答案生成

默认配置是 `generation.backend: extractive`，也就是抽取式回答。系统会选最相关证据片段，输出简短结论，并附上引用编号和提醒“以主管部门最新通知为准”。

如果改成 `generation.backend: llm`，可以调用 `Qwen/Qwen2-1.5B-Instruct`，并支持 `lora_adapter_path` 加载 LoRA adapter。Prompt 会要求模型：

- 严格依据检索证据回答。
- 证据不足时说明“当前知识库没有足够依据”。
- 关键结论后加 `[1]`、`[2]` 引用。

面试时要注意：

> 当前项目默认可在 CPU 上跑抽取式回答；Qwen2 和 LoRA 是可选生成增强，不是评测结果的唯一依赖。

## 6. 评估设计

评估问题位于 `data/campus_kb/eval_questions.jsonl`，共 81 条：

- 76 条可回答问题。
- 5 条拒答问题。
- 覆盖 `it`、`academic`、`student`、`finance`、`international`、`refusal` 六类。

每条评估样例包含：

- `question`：用户问题。
- `expected_doc_ids`：期望命中的文档 ID。
- `should_refuse`：是否应拒答。
- `category`：类别。

评估指标：

- `citation_hit_rate`：可回答问题中，引用至少命中一个期望文档的比例。
- `citation_recall_at_k`：所有期望文档中被引用命中的比例。
- `refusal_accuracy`：应该拒答的问题中，被正确拒答的比例。
- `false_refusal_rate`：本应回答但被拒答的比例。

当前结果：

- `citation_hit_rate`: 85.53%
- `citation_recall_at_k`: 82.14%
- `refusal_accuracy`: 80.00%
- `false_refusal_rate`: 14.47%
- 总评估问题数：81

这些指标说明系统大多数情况下能找对引用，但还有误拒和细分类别召回不足的问题，尤其 finance 样本较少，指标不稳定。

## 7. 面试时的项目讲述模板

可以按这个顺序讲：

1. 我做的是一个面向南京大学校园办事场景的文本 RAG 项目，目标是让学生能问 VPN、选课、成绩单、医保等问题，并得到带来源引用的回答。
2. 我把官方办事信息整理成 75 条 JSONL 知识库，保留标题、部门、来源、更新时间、标签和正文。
3. 检索上采用 BM25 和 Sentence-Transformer 双路召回，用 RRF 融合候选，再用 BERT Cross-Encoder 重排，提高引用命中率。
4. 生成上默认使用抽取式回答，也支持 Qwen2 和 LoRA adapter；Prompt 要求必须基于证据回答，证据不足时拒答。
5. 我构建了 81 条评估问题，统计引用命中率、Recall@K、拒答准确率和误拒率，当前整体引用命中率约 85.5%，Recall@K 约 82.1%，拒答准确率 80%。
6. 这个项目的价值在于把 RAG 从“能回答”推进到“可追溯、可拒答、可评估”，更适合政策和办事指南类场景。

## 8. 高频面试问题与参考答案

### Q1：这个项目主要解决什么问题？

答：它解决南京大学学生在校园办事中信息分散、检索困难、通用大模型容易幻觉的问题。系统把校园 IT、教务、学生服务、财务、出国境等材料整理成知识库，通过 RAG 检索证据并生成带引用的回答。如果知识库没有依据，系统会拒答，而不是编造。

### Q2：为什么这个项目适合用 RAG？

答：因为校园政策和办事流程强依赖外部知识，而且具有时效性。RAG 可以把回答绑定到知识库证据上，不需要把所有知识固化进模型参数。相比微调，RAG 更适合频繁更新的办事指南场景；知识变化时只需要更新 JSONL 和索引。

### Q3：你的知识库是怎么构建的？

答：我把常见校园办事场景整理成 JSONL 文档，每条文档包含 `id`、`title`、`department`、`source`、`updated_at`、`tags` 和 `text`。目前有 75 条，覆盖 VPN、统一身份认证、选课、成绩、推免、就业、医保、签证等高频问题。`source` 指向学校相关主管部门的官方入口或栏目页。

### Q4：为什么要保留 `source`、`department`、`updated_at` 这些字段？

答：这些字段用于增强可信度和可追溯性。回答时不仅给出文本，还能告诉用户来自哪个部门、哪个来源、更新时间是什么。对于校园政策类问答，出处比单纯答案更重要。

### Q5：为什么要做分块？

答：如果整篇文档直接编码，长文本中的局部信息会被稀释，检索不够精确。分块后，每个 chunk 更短、更聚焦，适合和用户问题做语义匹配。项目使用 420 字符窗口和 80 字符 overlap，既控制片段长度，又减少关键信息被切断。

### Q6：为什么用了 BM25 + 向量召回，而不是只用向量检索？

答：向量检索擅长语义相似，比如“校外访问数据库”和“VPN”可以匹配；但对专有名词、缩写、具体系统名不一定稳定。BM25 对关键词精确匹配强，比如 VPN、GPA、CET、Office。两者互补，所以我使用混合检索。

### Q7：BM25 的中文分词怎么做？

答：项目里用了轻量 tokenizer：英文数字按正则抽取，中文同时抽单字和 bigram。这样不依赖额外分词模型，也能覆盖一些中文短语匹配。缺点是没有真正理解中文词边界，后续可以接入 jieba 或专业中文分词。

### Q8：RRF 是什么，为什么用它？

答：RRF 是 Reciprocal Rank Fusion，按排序名次融合多个检索器，而不是直接融合原始分数。因为 BM25 分数和向量相似度不是同一尺度，直接相加不合理。RRF 用 `1 / (k + rank)` 的方式给排名靠前的候选更高分，简单稳定。

### Q9：Cross-Encoder 在项目里起什么作用？

答：BM25 和向量召回负责快速找候选，Cross-Encoder 负责精排。它把用户问题和候选片段一起输入模型，能捕捉更细粒度的匹配关系。项目只对前 10 个候选重排，避免全库 Cross-Encoder 带来的高成本。

### Q10：Bi-Encoder 和 Cross-Encoder 的区别是什么？

答：Bi-Encoder 分别编码问题和文档，适合大规模向量检索，速度快，可以预先建索引。Cross-Encoder 把问题和文档拼在一起编码，匹配更准，但不能预先为文档单独建索引，计算更慢。因此常见方案是 Bi-Encoder 召回，Cross-Encoder 重排。

### Q11：为什么默认生成端是抽取式，而不是直接用 Qwen？

答：默认抽取式回答更轻量、稳定、容易在 CPU 上运行，也更适合展示检索和引用能力。Qwen2/LoRA 是可选增强，用于更自然的语言组织。这个项目核心是检索、引用和拒答，不依赖大模型自由生成来制造亮点。

### Q12：Prompt 在项目中做了什么？

答：Prompt 用于约束生成端：要求模型作为校园办事指南助手，只能依据检索证据回答；证据不足时要明确说明没有足够依据；关键结论后用 `[1]`、`[2]` 标注引用。它的作用是降低幻觉和规范输出格式。

### Q13：LoRA 在这个项目中起什么作用？

答：LoRA 是可选的生成端适配手段，可以用少量校园办事问答样本让模型更适应办事指南口吻和引用格式。但知识本身仍然来自 RAG 检索，不应该依赖 LoRA 记住政策。面试时我会强调 LoRA 是格式和表达适配，不是替代知识库。

### Q14：系统如何判断应该拒答？

答：主要有三类条件：没有检索结果；Top1 命中知识库边界文档 `nju-support-unknown`；Top1 分数低于配置里的 `refusal_threshold`。拒答时会提示当前知识库没有足够依据，并建议查看主管部门官方通知。

### Q15：为什么要做拒答，而不是尽量回答？

答：校园政策类问答更看重准确和可依据。对于个人成绩查询、实时开放时间、账号状态、未公开政策等问题，系统没有权限或没有实时数据。如果强行回答会产生误导，所以拒答是安全能力的一部分。

### Q16：评估集是怎么设计的？

答：评估集有 81 条问题，每条标注期望命中的文档 ID、是否应拒答和类别。问题覆盖 IT、教务、学生服务、财务、国际事务和拒答边界。这样可以同时评估检索是否找对来源，以及系统是否能识别知识库外问题。

### Q17：为什么用 citation hit rate？

答：这个项目的关键不是生成文字和标准答案逐字一致，而是有没有引用到正确证据。`citation_hit_rate` 衡量每个可回答问题是否至少命中一个期望文档，更适合引用式 RAG。

### Q18：`citation_recall_at_k` 和 `citation_hit_rate` 有什么区别？

答：`citation_hit_rate` 只看一个问题是否至少命中一个期望文档；`citation_recall_at_k` 看所有期望文档中被命中的比例。如果一个问题期望两个文档，只命中一个，hit rate 算命中，但 recall 只算命中一半。

### Q19：当前评测结果怎么看？

答：整体引用命中率约 85.5%，Recall@K 约 82.1%，说明大多数问题能找到正确来源。拒答准确率 80%，说明知识库外问题大多能拒答。误拒率 14.5%，说明仍有一部分本应回答的问题被过滤掉，后续可以通过调阈值、扩充知识库、优化重排来改善。

### Q20：为什么 finance 类指标较低？

答：finance 评估样本只有 2 条，样本量很小，指标波动大；同时财务和后勤类问题在知识库中覆盖相对少，容易被相邻学生服务或信息化条目干扰。后续应补充财务处、后勤、水电缴费等更细的条目和评估问题。

### Q21：系统启动为什么可能需要访问 Hugging Face？

答：因为 Sentence-Transformer 和 Cross-Encoder 模型来自 Hugging Face。如果本地缓存缺少配置或权重，首次加载会联网下载。网络或代理不稳定时可能报 `RemoteProtocolError`。解决方式是提前缓存模型、设置镜像、或在缓存完整后启用离线模式。

### Q22：为什么要先运行 `build_campus_kb_index.py`？

答：它会读取知识库、分块、编码 embedding、构建 FAISS 索引，并保存 chunk 元数据。后续问答和评估可以直接加载索引，避免每次重新编码全部文档。

### Q23：FAISS 里存了什么？

答：FAISS 索引存的是 chunk embedding，而不是原文。原文、标题、来源等元数据保存在 `chunks.json`。检索时 FAISS 返回向量相似的 chunk 下标，再去元数据里取对应文本和来源。

### Q24：为什么使用 `IndexFlatIP`？

答：项目规模小，75 条文档分块后数量不大，Flat 暴力检索足够快，而且结果精确。embedding 做 L2 normalize 后，内积可以表示余弦相似度。后续如果知识库变大，可以换 IVF、HNSW 等近似索引。

### Q25：这个项目与普通 FAQ 系统有什么不同？

答：普通 FAQ 通常是问题到答案的匹配，覆盖范围受限；本项目是文档级 RAG，可以从办事指南片段中检索证据，并支持扩充知识库。它不仅返回答案，还返回引用、来源和检索分数，更利于追溯。

### Q26：这个项目与纯大模型问答有什么不同？

答：纯大模型依赖参数记忆，容易过期和幻觉；本项目把知识放在外部知识库中，回答基于检索证据，知识更新成本更低，也能明确拒答。

### Q27：这个项目和你的多模态 RAG 项目有什么区别？

答：校园 RAG 是纯文本办事指南，重点是文本召回、重排、引用和拒答；多模态 RAG 是图像问答，重点是 CLIP 图文检索和 Qwen2-VL 看图回答。两者都是 RAG，但检索对象、模态和评估重点不同。

### Q28：如果用户问“今天图书馆几点关”，系统应该怎么回答？

答：应该拒答或提示查看图书馆实时通知，因为开放时间可能随节假日和临时安排变化，知识库不具备实时性。项目中的拒答设计就是为了处理这类实时问题。

### Q29：如果用户问“帮我查我的 GPA”，系统应该怎么处理？

答：应该拒答。系统没有用户登录态，也不能访问个人教务数据。可以建议用户登录教务系统查询，但不能编造或请求隐私信息。

### Q30：如果知识库更新了，需要怎么处理？

答：在 JSONL 中追加或修改文档，确认字段完整，然后运行 `python build_campus_kb_index.py --force` 重建索引，再运行 `python evaluate_campus_kb.py` 检查指标变化。

### Q31：如何提升当前项目的效果？

答：可以从四方面优化：扩充和清洗知识库；引入更好的中文分词或领域词典；调节拒答阈值和重排候选数；构建更大、更均衡的评估集，尤其补充 finance、international 和拒答类样本。

### Q32：如果要上线给真实用户，还缺什么？

答：需要接入真实官方数据源和定期更新机制；增加权限和隐私边界；记录用户反馈和失败样例；做服务监控、缓存和限流；对答案增加更严格的引用校验；明确免责声明，提示以主管部门最新通知为准。

### Q33：有没有做消融实验？

答：项目配置支持消融，例如关闭 `retrieval.hybrid_enabled` 可以只用向量召回，关闭 `cross_encoder.enabled` 可以比较有无重排的效果。当前 README 中说明了 Cross-Encoder 可消融，但完整消融表可以作为后续补充。

### Q34：为什么没有用端到端微调一个模型？

答：因为校园办事知识会更新，微调模型成本高，而且更新后不容易追溯答案来源。RAG 更适合知识频繁变化和要求引用的场景。微调或 LoRA 可以用于格式适配，但不适合承载全部政策知识。

### Q35：如何减少幻觉？

答：第一，回答前必须检索证据；第二，Prompt 限制只能依据证据；第三，返回引用来源；第四，低置信度拒答；第五，默认抽取式回答减少自由生成。上线时还可以做引用一致性校验。

### Q36：为什么评估不使用 BLEU/ROUGE？

答：办事问答允许多种表述，BLEU/ROUGE 对中文开放式回答不稳定，而且不能反映证据是否正确。本项目更重视是否找到了正确文档和是否正确拒答，所以使用 citation hit rate、recall 和 refusal 指标。

### Q37：如果用户问题很口语化怎么办？

答：当前系统依赖向量召回处理口语表达，同时 BM25 保留关键词匹配。README 中也预留了查询改写 Prompt 的位置，后续可以在检索前用小模型把口语问题改写成更适合检索的标准问句。

### Q38：如何处理多个证据来源冲突？

答：当前系统会返回多个引用，但没有做冲突检测。真实上线时应按更新时间、主管部门权威性和政策层级排序，发现冲突时提示用户以最新官方通知为准，而不是强行合并。

### Q39：项目最大的不足是什么？

答：知识库规模仍然偏小，部分来源是官方入口而不是细粒度政策原文；评估集是人工构造，规模和覆盖还不够；拒答阈值是经验值，缺少系统调参；默认抽取式生成表达较模板化。这些都是后续可改进点。

### Q40：你在这个项目里主要做了什么？

答：我独立完成了知识库设计、文档分块、混合检索、RRF 融合、Cross-Encoder 重排、拒答逻辑、抽取式/LLM 可选生成、Gradio 演示和评估脚本，并整理了 75 条知识库和 81 条评估问题。

## 9. 可能的深入追问

### 追问 1：Cross-Encoder 分数和 RRF 分数尺度不同，拒答阈值怎么设？

参考回答：这是当前项目的一个工程权衡。开启 Cross-Encoder 后最终 `score` 会被替换为重排分数，因此阈值需要结合实际输出分布调节。更严谨的做法是把召回分和重排分分别保留，用验证集搜索阈值，或者训练一个二分类器判断 evidence 是否足够。

### 追问 2：为什么 `source` 有些是栏目入口，而不是具体文章？

参考回答：因为部分办事信息分布在需要登录或动态更新的系统中，项目不能依赖不可公开访问的页面。当前使用官方栏目入口保证可访问和可追溯。更好的做法是接入官方公开页面爬取或人工维护更细粒度的来源。

### 追问 3：BM25 的中文 tokenizer 会不会太粗糙？

参考回答：会，这是轻量实现。它通过单字和 bigram 保留一定中文匹配能力，但不如正式中文分词。项目规模较小时足够演示混合检索思想，后续可以引入 jieba、THULAC 或领域词典，并对比评估指标。

### 追问 4：为什么拒答准确率不是 100%？

参考回答：拒答本身是检索式系统的难点。部分边界问题可能会检索到相似但不完全相关的文档，导致系统给出引用。可以通过扩充拒答样例、调整阈值、增加“问题类型分类器”或训练 entailment 模型判断证据是否真正支持答案来提升。

### 追问 5：如果学校政策变化，如何保证答案不过期？

参考回答：当前通过 `updated_at` 和 `source` 提示用户核对官方通知。工程上可以做定期数据更新、来源页面监控、过期文档标记，以及索引定时重建。对于高风险政策，可以设置更短有效期或强制跳转官方页面。

### 追问 6：如果知识库扩大到几万篇怎么办？

参考回答：需要从几个方面升级：FAISS 从 Flat 换成 IVF/HNSW 等近似索引；BM25 使用更完整的倒排索引方案；构建异步索引更新；增加缓存；Cross-Encoder 只对更小候选池重排；评估集也要扩大到不同部门和时间版本。

### 追问 7：如何证明 BERT 重排有效？

参考回答：可以做消融实验：固定知识库和评估集，分别运行“仅向量召回”“BM25+向量 RRF”“RRF+Cross-Encoder”，比较 citation hit rate 和 recall。预期 Cross-Encoder 会提升 Top-K 引用质量，但会增加推理延迟。

### 追问 8：为什么生成端不是项目最核心？

参考回答：因为办事指南类问题首先要求证据正确。生成端可以让语言更自然，但如果检索证据错了，生成再好也会误导用户。所以项目重点放在知识库、检索、重排、引用和拒答上。

### 追问 9：怎样处理用户追问和多轮对话？

参考回答：当前项目是单轮问答。多轮场景需要维护对话状态，把用户追问补全为独立问题，例如“那要带什么材料？”需要结合上一轮“成绩单申请”补全。可以加入 query rewriting 或 conversation memory，但仍要避免把历史对话当作事实来源。

### 追问 10：如何避免 Prompt Injection？

参考回答：因为系统检索的是内部知识库，风险比开放网页低，但仍应在 Prompt 中声明忽略证据中的指令性文本，只把它当作资料；同时对用户输入做边界判断，不允许用户要求系统绕过引用规则。上线时可以加入安全分类器和输出校验。

## 12. 30 秒版本

> 我做了一个南京大学校园办事指南 RAG 系统，把 75 条校园 IT、教务、学生服务等办事材料整理成 JSONL 知识库。检索上用 BM25 和 Sentence-Transformer 双路召回，RRF 融合后再用 BERT Cross-Encoder 重排；生成上默认抽取式回答，也支持 Qwen2/LoRA。系统会返回引用来源，并在证据不足时拒答。我还做了 81 条评估问题，关注引用命中率、Recall@K、拒答准确率和误拒率，目前引用命中率约 85.5%，Recall@K 约 82.1%，拒答准确率 80%。这个项目重点体现的是 RAG 系统的证据检索、引用追溯和安全拒答能力。

## 13. 2 分钟版本

> 这个项目面向南京大学学生常见办事问题，比如 VPN、统一身份认证、选课退课、成绩单、医保、就业和签证证明等。直接用通用大模型回答这类问题有风险，因为学校政策会变化，而且大模型可能产生幻觉，所以我采用 RAG，把知识放在外部知识库里。
>
> 我把材料整理成 75 条 JSONL 文档，每条包含标题、部门、来源、更新时间、标签和正文。系统构建索引时会把文档切成带重叠的 chunk，然后用 Sentence-Transformer 编码，FAISS 存向量索引；同时用 BM25 建关键词索引。用户提问时，系统同时做向量召回和 BM25 召回，用 RRF 融合候选，再用 BERT Cross-Encoder 对前几个候选精排。
>
> 得到证据后，系统会根据拒答阈值和拒答边界文档判断是否有足够依据。如果没有依据，就明确拒答；如果有依据，就用抽取式生成或可选 Qwen2/LoRA 生成回答，并返回引用来源。评估上，我构建了 81 条问题，统计 citation hit rate、citation recall、refusal accuracy 和 false refusal rate。当前引用命中率约 85.5%，Recall@K 约 82.1%，说明大多数问题能找到正确依据。这个项目的核心价值是让校园办事问答变得可追溯、可拒答、可评估。

## 14. 如果被问“项目亮点”

可以回答：

- 不是简单套壳大模型，而是完整实现了知识库、索引、混合检索、重排、拒答、引用和评估闭环。
- 混合检索兼顾语义召回和关键词精确匹配，适合校园办事中的专有名词。
- Cross-Encoder 重排提高引用准确性。
- 拒答机制适合政策和办事类场景，能避免回答实时、隐私和知识库外问题。
- 评估不是只看主观效果，而是用引用命中率、Recall@K、拒答准确率和误拒率量化。

## 15. 如果被问“项目不足”

可以回答：

- 知识库规模还不大，只有 75 条，部分条目来源是官方入口页而不是细粒度原文。
- 评估集 81 条，能反映基本趋势，但还不够大，finance 和 international 类样本较少。
- 中文 BM25 tokenizer 比较轻量，没有引入专业分词。
- 拒答阈值是经验设置，缺少系统阈值搜索。
- 默认抽取式回答稳定但不够自然；LLM 生成需要更强的引用一致性校验。

## 16. 后续改进方向

- 扩充知识库，接入更多官方公开页面，并定期更新。
- 增加更均衡的评估集，尤其补充财务、后勤、国际交流和拒答边界问题。
- 做正式消融实验，比较仅向量、仅 BM25、RRF、Cross-Encoder 的指标变化。
- 引入中文分词和领域词典，改善 BM25。
- 对拒答阈值做验证集搜索，降低误拒率。
- 增加答案引用一致性检查，防止生成内容超出证据。
- 支持多轮问答中的查询改写。
- 增加日志和用户反馈闭环，用真实失败样例迭代知识库。
